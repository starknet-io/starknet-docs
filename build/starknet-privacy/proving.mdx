Proofs are generated by **executing Cairo in a simulated Starknet environment** tied to a recent block, then proving that execution with **Stwo**—the same prover family used for Starknet block proving.

## Why this shape

| Benefit | Explanation |
| ------- | ----------- |
| **Native storage** | Notes live in **contract storage**; the proof reads anchored state instead of maintaining a separate Merkle tree of commitments like Zcash. |
| **Signature abstraction** | Inside the VM, the proof calls the user’s **account contract** `is_valid_signature` — multisig, smart wallets, social recovery all work if they implement [SNIP-6](https://github.com/starknet-io/SNIPs/blob/main/SNIPS/snip-6.md). |
| **Single codebase** | Hashing, encryption, and compilation logic are shared between **prover** and **on-chain verifier** in **Cairo**. |

## Viewing vs signing

- **Viewing key** — discover and decrypt; needed to form nullifiers when spending.
- **Account private key / account logic** — authorization to spend; the auditor must **not** be able to sign on behalf of users.

## Anchoring

Proofs reference storage from a **recent** block; **WriteOnce** on-chain ensures cells the proof relied on weren’t changed in a way that breaks soundness for that note path.

Read more: [IACR 2026/474](https://eprint.iacr.org/2026/474.pdf) §8.
