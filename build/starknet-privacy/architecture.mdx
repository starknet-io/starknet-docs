The design has three pillars (see [IACR 2026/474](https://eprint.iacr.org/2026/474.pdf) §3).

## 1. Privacy pool contract

A canonical Starknet contract holds user assets and stores:

- **Encrypted notes** and **nullifiers**
- Metadata for **note discovery** (channels, subchannels, markers)
- **Write-once** storage: cells are written once and never overwritten (immutability for privacy and proof consistency)

The contract verifies **ZK proofs** and executes **server-side actions** (writes, channel appends, token transfers).

## 2. Client-side proving

Users build **client-side actions** and generate a **Stwo** proof over **Cairo** that simulates execution in a **virtual Starknet** environment anchored to a recent block. The proof attests:

- Valid **account signature** ([SNIP-6](https://github.com/starknet-io/SNIPs/blob/main/SNIPS/snip-6.md) `is_valid_signature` in the simulated environment)
- Correct note existence, nullifiers, encryption, and **balance conservation**

## 3. Compliance framework

On registration, each user stores a **public viewing key** and an **encrypted private viewing key** for a designated **auditing entity**. That enables selective tracing under regulatory process without decrypting unrelated users’ data.

## Data model (preview)

- **Notes** — owner, token, amount (amount often encrypted at rest).
- **Channels** — unidirectional sender → recipient; shared **channel key** for discovery and encryption.
- **Subchannels** — per-channel, per-token lanes so one channel can carry many assets without extra channel opens.
