**Starknet Privacy** is a protocol for private asset transfers on Starknet: sender, receiver, and amounts are hidden from outside observers, while validity is enforced with client-side zero-knowledge proofs (Stwo / Cairo). A compliance layer allows an auditing entity to trace activity when legally required, without exposing unrelated users.

## Why Starknet Privacy?

If you care about confidential transfers, you’ve probably seen mixers (fixed amounts, one-time use), or full privacy chains (trial-decrypt everything, or separate ecosystem). Starknet Privacy takes a different path:

- **Stay on Starknet** — No new chain. You keep access to existing DeFi, accounts, and tooling. Privacy is a layer on the L2 you’re already using.
- **Arbitrary amounts, reusable notes** — No fixed denominations. Notes can move many times inside the pool before withdrawal, so you’re not forced into a single deposit/withdraw pattern.
- **Discovery that scales with you** — Recipients find incoming notes in time proportional to *their* channels and activity, not the entire pool. No scanning every transaction like in classic note-based designs.
- **Compliance by design** — Viewing keys can be escrowed to an auditor so that *only* the subject of a lawful request is traceable. Everyone else stays private. That makes it easier for regulated entities to adopt without “all or nothing” transparency.
- **One account for everything** — Spend authority uses your existing Starknet account (multisig, smart wallet, etc.). You don’t manage a separate “privacy key” for signing—only a viewing key for reading and decrypting.

So: confidential transfers, efficient discovery, and selective disclosure, without leaving Starknet or locking you into a one-off mixer.

## What you get

| Property | Summary |
| -------- | ------- |
| **Private transfers** | Sender, receiver, and amount hidden from observers; transacting parties and authorized auditors can see details. |
| **Compliance** | Designated auditor can selectively trace history under legitimate regulatory request. |
| **Efficient discovery** | Recipients find notes in time scaling with *their* activity, not total pool activity. |
| **Multi-asset** | One pool supports multiple ERC-20 token types. |
| **DeFi** | Anonymous interaction with existing Starknet DeFi (e.g. AMM swaps) via open notes and atomic flows. |
| **Account abstraction** | Spend authority uses your Starknet account (multisig, smart wallets, etc.); viewing key handles read/decrypt paths. |

## How it works (high level)

1. Users **deposit** tokens into a **privacy pool** contract.
2. Assets inside the pool are **notes** (encrypted on-chain); spending publishes a **nullifier** so notes cannot be double-spent.
3. Each transaction includes a **ZK proof** (generated off-chain with Stwo) that the contract verifies.
4. **Paymasters** can submit transactions so fee payment does not trivially identify the user on-chain.

For protocol details, continue with [Architecture](/build/starknet-privacy/architecture) and the [paper](https://eprint.iacr.org/2026/474.pdf).
