## Overview

A transaction is a signed request to change something on Starknet.

For example, a transaction can call a smart contract, register a new contract class, or deploy a new account. After a transaction is sent, Starknet validates it, tries to execute it, assigns it [statuses](#transaction_statuses), and if it is executed, returns a [receipt](#transaction_receipt) describing what happened.

## Transaction types

Starknet supports the following types of transactions:

|                  |                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                            |
| ---------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `DECLARE`        | Introduces a new contract class to Starknet, enabling other contracts to either deploy instances of it or use it via a library call<Tip>         For more information about contract classes and instances, see the Cairo Book.   </Tip>                                                                                                                                                                                                                                                                                   |
| `INVOKE`         | Invokes a function in an existing contract instance, undergoing validation and execution initiated by the caller account's [`__validate__ and __execute__ functions`](/learn/protocol/accounts/#starknets_account_structure) (respectively).<Note>         The validation stage verifies that the caller account approves the transaction. Because an account's \__validate_\_ and \__execute_\_ functions can contain any logic, the caller account ultimately determines how to handle the INVOKE transaction.   </Note> |
| `DEPLOY_ACCOUNT` | Deploys a new account contract to Starknet.<Tip>         For more information about deploying a new account, see Accounts.   </Tip>                                                                                                                                                                                                                                                                                                                                                                                        |

<Note>
  The [`L1HandlerTransaction type`](/learn/protocol/messaging/#l1_handler_transaction) is also a valid transaction type within Starknet, but it cannot be broadcast through the Starknet API like other transaction types, as it was specifically designed for internal Starknet operations, particularly handling messages from L1 to L2.

  The `DEPLOY` transaction type existed in Starknet for deploying account before being replaced by `DEPLOY_ACCOUNT`. `DEPLOY` only had one version (v0) and is now unsupported by Starknet.
</Note>

Each transaction type is versioned. A version changes when the transaction format changes, for example when fields are added, removed, or interpreted differently.

Today, Starknet accepts only RPC 0.8-compatible `v3` transactions. Older versions are unsupported.

| Type             | Current version | Unsupported versions |
| ---------------- | --------------- | -------------------- |
| `INVOKE`         | v3              | v1, v0               |
| `DECLARE`        | v3              | v2, v1, v0           |
| `DEPLOY_ACCOUNT` | v3              | v1                   |

<Warning>
  Do not submit a transaction that uses an unsupported version. Since Starknet v0.14.0, only RPC 0.8-compatible `v3` transactions are accepted by the network.
</Warning>

## How transaction versions evolved

The current `v3`-only model is the result of several protocol changes over time:

| Starknet version | Transaction change |
| ---------------- | ------------------ |
| `v0.10.0`        | Introduced `INVOKE v1` and `DECLARE v1`, added protocol-level nonces, and separated validation from execution for account contracts. |
| `v0.10.1`        | Introduced `DEPLOY_ACCOUNT`, replacing the old `DEPLOY` transaction for account deployment. |
| `v0.11.0`        | Introduced `DECLARE v2` to support Sierra and Cairo 1 contract classes. |
| `v0.13.0`        | Introduced `v3` transactions, including STRK-denominated fees and reserved fields for future features such as paymasters and data availability modes. |
| `v0.13.4` / `v0.13.5` | Added the three-resource fee model and the RPC 0.8-compatible `v3` format used today. |
| `v0.14.0`        | Stopped accepting `v0`, `v1`, `v2`, and old non-RPC-0.8-compatible `v3` transactions; introduced mempools, fee-market ordering, and the `CANDIDATE` / `PRE_CONFIRMED` statuses. |

<Tip>
  For the exact fields and hash calculations of each transaction version, see [Transaction fields and hash calculations reference](/learn/cheatsheets/transactions-reference/).
</Tip>

## Transaction lifecycle

The high-level steps in the Starknet transaction lifecycle are as follows:

1. **Submission:** A wallet, app, or service sends the transaction through a full node or provider to a sequencer gateway.
2. **Mempool admission:** The sequencer checks whether the transaction can enter its mempool. For example, it verifies basic validity, nonce rules, and fee bounds. If the transaction is invalid, it does not proceed.
3. **Block building and validation:** When building a block, the sequencer revalidates the transaction against the latest relevant state before execution.
4. **Execution:** The sequencer executes the transaction. If execution fails, the transaction can still be included in the block as `REVERTED`.
5. **Finalization and proving:** Once transactions are included in a finalized block, the [Starknet operating system](/learn/protocol/snos/) proves the block execution, and the result is eventually verified on L1.

### Mempools

Up to and including Starknet v0.13.5, transactions were effectively ordered in a first-in-first-out (FIFO) queue. Starting from Starknet v0.14.0, each sequencer maintains its own mempool. This means the order in which transactions arrive is no longer necessarily the order in which they are included in blocks.

Starting from Starknet v0.14.0, mempools affect transaction processing in the following ways:

- **Nonces**: Transactions with a nonce greater than the sender's current nonce + $x$ will be rejected by the mempool, where:
  - A sender's current nonce is the sender's nonce in the latest finalized state at the time the transaction is received
  - $x$ is 200 for `INVOKE` and `DEPLOY_ACCOUNT` transactions, and 0 for `DECLARE` transactions
- **Fee escalation**: A transaction in the mempool can be replaced by another transaction from the same account with the same nonce if both `tip` and the `l2_gas` `max_price_per_unit` are increased by at least 10%. If the original transaction used zero tip, it can still be replaced by another transaction with zero tip.

  <Note>
    There are unlikely edge cases in which a transaction is not replaced despite submitting a valid transaction to replace it, such as the existing transaction entering a block while the new one is still processed in the gateway.
  </Note>
- **Transaction time-to-live (TTL)**: Transactions that stay in the mempool for too long without being included are evicted. The current TTL is 5 minutes.
- **`DECLARE` time-to-mature (TTM)**: Before entering the mempool, `DECLARE` transactions spend time in a waiting room. During that period they cannot be replaced, and another transaction from the same account with the same nonce is rejected. The TTM is smaller than the TTL.

## Transaction statuses

Since Starknet v0.14.0, Starknet uses the following transaction statuses:

| Type          | Status           | Description                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| ------------- | ---------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Finality**  | `NOT_RECEIVED`   | The transaction is not yet known to any sequencer.                                                                                                                                                                                                                                                                                                                                                                                                                                |
|               | `RECEIVED`       | A full node has successfully submitted the transaction to a sequencer.<Note>         As there is currently no P2P protocol between full nodes for updating on each other about received transactions, querying a different node than the one that submitted the transaction for its status will result in a Transaction hash not found error.<br /><br />Therefore, relying on RECEIVED statuses requires initiating sticky HTTP sessions with your full node provider.   </Note> |
|               | `CANDIDATE`      | The transaction was written to the feeder gateway's storage by a sequencer.<Note>         As the transaction hasn't been executed yet, no execution information is available and only the transaction hash is written to the feeder gateway's storage.    </Note>                                                                                                                                                                                                                 |
|               | `PRE_CONFIRMED`  | The transaction was successfully executed by a sequencer and its receipt was written to the feeder gateway's storage.                                                                                                                                                                                                                                                                                                                                                             |
|               | `ACCEPTED_ON_L2` | The transaction was included in a block finalized by the consensus protocol.                                                                                                                                                                                                                                                                                                                                                                                                      |
|               | `ACCEPTED_ON_L1` | The Starknet state on L1 moved to a block height which is greater than or equal to the height of the block containing the transaction.                                                                                                                                                                                                                                                                                                                                            |
| **Execution** | `REVERTED`       | The transaction failed during execution by a sequencer, and was included in a block as reverted.<Note>         Since the DEPLOY_ACCOUNT and DECLARE transactions don't have an execution phase, they cannot be reverted.   </Note>                                                                                                                                                                                                                                                |
|               | `SUCCEEDED`      | The transaction was successfully executed by a sequencer and is included in a block.                                                                                                                                                                                                                                                                                                                                                                                              |

<Note>
  Before Starknet v0.14.0, readers may encounter older references to a `PENDING` block or `PENDING` transaction semantics. In the current model, `PRE_CONFIRMED` means the transaction was executed and exposed by the sequencer before consensus finalized the block, while `ACCEPTED_ON_L2` means the transaction is already in a consensus-finalized block.
</Note>

### State implications of reverted transactions

When a transaction is marked as `REVERTED`, the following state implications occur:

|                   |                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| ----------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Nonce increases   | The nonce value for the account of the failed transaction iterates despite the failure.                                                                                                                                                                                                                                                                                                                                                                                                                                                                                              |
| Fee charge        | The sequencer charges a fee for the execution of the transaction up to the point of failure, and a respective `Transfer` event is emitted.                                                                                                                                                                                                                                                                                                                                                                                                                                           |
| Partial reversion | All changes that occurred during the validation stage are not reverted. However, all changes that occurred during the execution stage are reverted, including all messages to L1 or any events that were emitted during this stage. Events might still be emitted from the validation stage or the fee charge stage.                                                                                                                                                                                                                                                                 |
| Fee calculation   | The fee charged for `REVERTED` transactions is the smaller of the following two values:<br /><br />• The maximum fee that the user is willing to pay, either `max_fee` (pre-v3 transactions) or $\texttt{max\_amount} \cdot \texttt{max\_price\_per\_unit}$ (v3 transactions).<br /><br />• The total consumed resources.<Note>         Consumed resources are resources used for the execution of the transaction up to the point of failure, including Cairo steps, builtins, syscalls, L1 messages, events, and state diffs during the validation and execution stages.   </Note> |

## Transaction receipt

A transaction receipt can be obtained by using the Starknet API's `starknet_getTransactionReceipt` endpoint and contains the following fields:

|                      |                                                                  |
| -------------------- | ---------------------------------------------------------------- |
| `transaction_hash`   | The hash of the transaction.                                     |
| `actual_fee`         | The actual fee paid for the transaction.                         |
| `finality_status`    | The finality status of the transaction.                          |
| `execution_status`   | The execution status of the transaction.                         |
| `block_hash`         | The hash of the block that includes the transaction              |
| `block_number`       | The sequential number of the block that includes the transaction |
| `messages_sent`      | A list of messages sent to L1.                                   |
| `events`             | The events emitted.                                              |
| `execution_resource` | A summary of the execution resources used by the transaction.    |
| `type`               | The type of the transaction.                                     |
